## 🔴 CRÍTICO - Problemas Graves

### 1. **OTP NO SE PERSISTE EN BD**

**Ubicación**: `/api/otp/send` y `/api/verify/sms`

**Problema**:

- `POST /api/otp/send`: Genera OTP de 6 dígitos pero NO lo guarda en `OtpToken`
- `POST /api/verify/sms`: Valida contra hardcoded "123456" (dev) o "??????" (prod)
- **Resultado**: No hay validación real de OTP en producción

**Impacta**:

- ❌ ATHLETE + SMS (en onboarding)
- ❌ COACH + Phone (en onboarding Step -1)

**Solución necesaria**:

```typescript
// En /api/otp/send: Guardar OTP en BD
await prisma.otpToken.create({
  data: {
    userId: session.user.id,
    code: otp,
    type: OtpType.VERIFICATION, // o crear nuevo tipo
    expiresAt: new Date(Date.now() + 10 * 60 * 1000),
  },
})

// En /api/verify/sms: Validar contra BD
const validToken = await prisma.otpToken.findFirst({
  where: {
    userId: session.user.id,
    code: otp,
    expiresAt: { gt: new Date() },
  },
})
```

### 2. **COACH: Phone Verification NO ENVIADA**

**Ubicación**: `/api/register` (POST)

**Problema**:

- Coach se registra con teléfono obligatorio
- NO se envía SMS automático de verificación
- Coach debe esperar a onboarding para intentar verificar

**Impacta**:

- ❌ COACH + SMS flow durante registro
- ❌ No hay SMS enviado automáticamente

**Esperado**:

1. Coach se registra con phone
2. **Auto-envía SMS** con OTP (o link/verificación)
3. Coach verifica en onboarding Step -1
4. Avanza a Step 0

**Solución necesaria**:

```typescript
// En /api/register POST, después de crear Coach profile:
if (userRole === Role.COACH) {
  // ... crear profile ...

  // AGREGAR: Enviar OTP de phone verification
  const otp = Math.floor(Math.random() * 1000000)
    .toString()
    .padStart(6, '0')
  const expiresAt = new Date(Date.now() + 10 * 60 * 1000)

  // Guardar OTP en BD
  await prisma.otpToken.create({
    data: {
      userId: user.id,
      code: otp,
      type: 'PHONE_VERIFICATION', // enum new
      expiresAt,
    },
  })

  // Enviar SMS
  await sendSms({
    to: phone,
    message: `Tu código de verificación NEXUM es: ${otp}. Válido por 10 minutos.`,
  })
}
```

### 3. **ATHLETE + SMS: OTP NO SE ENVÍA EN REGISTRO**

**Ubicación**: `/app/register/page.tsx` (frontend)

**Problema**:

- Athlete elige SMS en registro
- NO se envía OTP durante registro
- OTP solo se envía en onboarding Step -1
- Usuario no entiende por qué no tiene OTP todavía

**Impacta**:

- 🟡 UX confusa (elige SMS pero no lo recibe)

**Opciones**:

1. **Enviar OTP en registro** (mejor UX)
   - Después de crear User, envía OTP a phone
   - En onboarding Step -1, usuario solo ingresa OTP (ya está en su teléfono)

2. **Explicar en UI** (status quo)
   - En register, mostrar texto: "Recibirás el OTP en el siguiente paso"
   - En onboarding, enviar OTP

**Recomendación**: Opción 1 (mejor flujo)

## 🟡 IMPORTANTE - Inconsistencias de Diseño

### 4. **Flujo SMSdistinto para Coach y Athlete**

**COACH SMS**:

1. Registro: crea perfil con phone
2. Onboarding Step -1: debería enviar OTP
3. Valida OTP
4. Avanza

**ATHLETE SMS**:

1. Registro: elige SMS, se crea User
2. Onboarding Step -1: envía OTP desde aquí
3. Valida OTP
4. Se crea Athlete profile con phoneVerified=true
5. Avanza a Step 0

**Problema**: Flujos diferentes, confuso

**Recomendación**: Unificar - ambos envían OTP en onboarding Step -1
